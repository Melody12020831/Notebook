敏捷开发内涵 必考

核心问题：如何从瀑布式的预先规划转向迭代式的持续交付——以应变能力取代对计划的盲从。

敏捷宣言四大价值观

  1. 个体和互动 高于 流程和工具
  2. 可工作的软件 高于 详尽的文档
  3. 客户合作 高于 合同谈判
  4. 响应变化 高于 遵循计划

「虽然右项也有价值，但我们更重视左项的价值。」

Scrum 框架 必考

角色职责
Product Owner（产品负责人）定义需求、排序 Backlog、最大化价值
Scrum Master过程教练——移除障碍、促进 Scrum 实践
Development Team自组织、跨职能——3-9 人

核心仪式：Sprint Planning → Daily Scrum → Sprint Review → Sprint Retrospective

CI/CD 必考

  1. 持续集成 (CI)：每次提交自动构建+测试——快速发现集成问题
  2. 持续交付 (CD)：CI 通过后自动部署到类生产环境——随时可手动发布
  3. 持续部署 (CD)：通过测试后自动部署到生产环境——全自动化发布

DevOps 必考

DevOps = 开发 (Dev) + 运维 (Ops) 的文化融合——打破「开发丢代码给运维」的墙。

  • CAMS 模型：Culture（文化）+ Automation（自动化）+ Measurement（度量）+ Sharing（共享）
  • Infrastructure as Code (IaC)：用代码定义和管理基础设施（Terraform/Ansible）
  • Monitoring & Observability：生产环境的持续监控和反馈
易错：敏捷不是「不写文档」「不做计划」——而是做「刚好够的文档和计划」。Scrum 不是敏捷的全部——XP、Kanban 也是敏捷实践。CI/CD ≠ DevOps——CI/CD 是技术管道，DevOps 是文化+技术+流程的融合。
