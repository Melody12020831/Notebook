25.1 WebApp 测试质量维度

核心问题：WebApp 测试远不止功能测试——需要从多个质量维度全面验证，包括内容、功能、结构、可用性、导航性、性能、兼容性、互操作性、安全性。

内容测试维度 必考

内容在两个层面评估：

层面检查内容
语法层面 (Syntactic)拼写、标点、语法——针对文本文档
语义层面 (Semantic)信息正确性、一致性（跨内容对象）、无歧义

内容测试三目标：①发现文字/图形/媒体的语法错误；②发现语义错误（信息准确性或完整性）；③发现内容组织结构错误。

功能与结构测试 重点

  • 功能 (Function)：测试正确性、稳定性和对实现标准的符合性（如 Java/XML 语言标准）
  • 结构 (Structure)：确保正确传递内容且功能可扩展——新内容/功能可被支持

可用性与导航性 重点

  • 可用性 (Usability)：每类用户都能被界面支持、能学习和应用所有导航语法和语义
  • 导航性 (Navigability)：测试所有导航机制——发现死链接、错误链接、不当链接

性能与安全性

  • 性能 (Performance)：在各种操作条件/配置/负载下测试响应性和极端负载下的行为
  • 兼容性 (Compatibility)：在不同主机配置（客户端+服务器端）执行 WebApp，发现特定配置的错误
  • 互操作性 (Interoperability)：确保与其他应用/数据库正确对接
  • 安全性 (Security)：评估潜在漏洞并尝试利用——任何成功渗透都视为安全失败
易错：WebApp 测试有 9 个质量维度，功能测试只是其中之一。考试常考各个维度的具体测试内容——尤其注意区分「可用性」和「导航性」：可用性关注用户能否学会使用，导航性关注链接是否正确。

25.2-25.3 测试过程与内容测试 重点

内容语义评估清单

  • 信息是否事实准确？
  • 是否简洁切题？
  • 布局是否易于理解？
  • 嵌入信息是否容易找到？
  • 外部来源是否有适当引用？
  • 内容是否内部一致且与其他对象一致？
  • 内容是否冒犯、误导或引发诉讼？
  • 是否侵犯版权或商标？
  • 内部链接是否正确？
  • 美学风格是否与界面冲突？

25.4 用户界面测试 必考

界面测试层次

  1. 界面特性测试——确保设计规则、美学和视觉内容无误可用
  2. 个体界面机制测试——类比单元测试，逐个测试每种界面机制
  3. 用例上下文中的界面机制测试——针对特定用户类别的用例/NSU
  4. 完整界面测试——针对选定用例/NSU 发现界面语义错误
  5. 多环境兼容测试——在不同浏览器中测试

需测试的界面机制 必考

机制说明
链接 (Links)导航到其他内容或功能的机制
表单 (Forms)含空白字段的结构化文档，输入用于 WebApp 功能
客户端脚本JavaScript 等处理表单输入或用户交互
动态 HTML通过脚本/CSS 在客户端操纵内容对象
弹出窗口无用户交互自动弹出的小窗口
CGI 脚本Web 服务器与用户动态交互的标准方法
流媒体内容服务器主动推送数据（"Push" 技术）
Cookies服务器发送、浏览器存储的数据块
应用特定机制购物车、信用卡处理、运费计算器等宏机制

可用性测试步骤

  1. 定义可用性测试类别和每个类别的目标
  2. 设计能评估每个目标的测试
  3. 选择参与测试的人员
  4. 记录参与者与 WebApp 交互过程
  5. 开发评估可用性的机制

三个抽象层次：个体界面机制 → 完整网页 → 整个 WebApp

25.5-25.8 其他测试领域

导航测试 高频

需测试的导航机制：导航链接（内部/外部/锚点）、重定向（不存在 URL 的处理）、书签（页面标题可提取）、框架和框架集、站点地图（每项链接正确）、内部搜索引擎（准确性和完整性）。

配置测试维度

服务器端：OS 兼容性、系统文件正确创建、安全措施不影响性能、分布式配置测试、数据库集成与版本敏感性、脚本执行、管理错误影响。

客户端：硬件（CPU/内存/存储/打印）、OS（Linux/Mac/Windows/移动OS）、浏览器（IE/Chrome/Firefox/Safari）、界面组件（ActiveX/Java applets）、插件（QuickTime/RealPlayer）、连接方式（cable/DSL/WiFi/4G/5G）。

关键：配置变量数量必须缩减到可管理范围——不可能测试所有组合。

安全测试

探测客户端环境（浏览器/邮件程序漏洞）、网络通信（客户端-服务器数据传输）、服务器端环境（DDoS 攻击、恶意脚本）的漏洞。

25.9 性能测试 必考

负载测试核心指标

P = N × T × D

  • N = 并发用户数
  • T = 每单位时间在线事务数
  • D = 每事务服务端处理的数据负载
  • P = 整体吞吐量 (Throughput)

压力测试关键问题

  • 系统是「温和退化」还是直接关闭？
  • 用户是否能感知到服务器不可达？
  • 请求是否排队？容量恢复后队列是否清空？
  • 超容量时事务是否丢失？数据完整性是否受影响？
  • 什么 N/T/D 值导致失败？失败如何表现？
  • 恢复在线需要多长时间？
  • 容量达 80-90% 时是否暂停某些功能？
易错：负载测试 (Load Testing) 和压力测试 (Stress Testing) 不是一回事。负载测试 = 确定系统在不同负载条件下的响应；压力测试 = 找到系统的崩溃点并观察崩溃行为。
