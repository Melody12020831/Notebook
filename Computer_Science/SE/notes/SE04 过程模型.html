4.1 规约式过程模型

核心问题：规约式模型倡导有序、结构化的软件工程方法——但在一个以变化驱动的软件世界中，这种结构化的方法是否仍然合适？

瀑布模型 (Waterfall Model) 必考

经典生命周期，按顺序执行：沟通 → 计划 → 建模 → 构建 → 部署

阶段内容
沟通项目启动、需求收集
计划估算、进度安排、跟踪
建模分析和设计
构建编码和测试
部署交付、支持、反馈

缺点：实际项目很少遵循顺序流；客户通常不能一次性明确所有需求；可运行版本要到项目后期才可用。

V 模型 重点

瀑布模型的变体——强调测试与开发的对应关系：需求建模 ↔ 验收测试、架构设计 ↔ 系统测试、组件设计 ↔ 集成测试、代码生成 ↔ 单元测试。

易错：瀑布模型不是完全不能用，而是有前提条件——需求稳定、明确、不太可能变更。判断常见陷阱：一说到瀑布就全盘否定。

增量过程模型 必考

每一个增量交付一个可运行的功能子集。第一个增量通常是核心产品 (Core Product)，后续增量增加更多特性和功能。更好地利用资源。

演化过程模型 — 原型开发 重点

原型必须被丢弃 (must be thrown away)。适用于客户有明确需求但不清楚细节的场景。步骤：沟通 → 快速计划 → 快速设计建模 → 原型构建 → 部署交付与反馈。

易错：原型不是最终产品，是「用过即弃」的沟通工具。常见错误：把原型当最终产品交付。

螺旋模型 (Spiral Model) 必考

以风险分析为核心驱动力。每个螺旋周期包含：确定目标/方案/约束 → 评估方案/识别并解决风险 → 开发验证下一级产品 → 计划下一阶段。

核心特征：综合了原型开发的迭代性质和瀑布模型的可控性和系统性。

并发开发模型 高频

定义活动网络而非线性事件序列。通过状态转换事件触发各活动中的状态变化。尤其适合客户/服务器应用。

4.2 专用过程模型

  • 构件化开发 (Component-Based Development)：以复用为目标的开发过程
  • 形式化方法 (Formal Methods)：强调需求的数学规格说明
  • 面向方面软件开发 (AOSD)：定义、规格、设计和构建方面的过程方法

4.3 统一过程 (Unified Process) 必考

「用例驱动、架构为中心、迭代和增量」的软件过程，与 UML 紧密对齐。

UP 阶段与工作产品

阶段核心工作产品
初始 (Inception)愿景文档、初始用例模型、项目术语表、初始业务用例、初始风险评估、项目计划
细化 (Elaboration)用例模型、需求（功能/非功能）、分析模型、软件架构描述、可执行架构原型、修订风险列表
构建 (Construction)设计模型、软件组件、集成软件增量、测试计划和用例、支持文档
移交 (Transition)交付软件增量、Beta 测试报告、用户反馈

4.4 个人与团队过程模型

PSP (Personal Software Process)：五个框架活动——计划、高层设计、高层设计评审、开发、事后分析。强调尽早识别错误并理解错误类型。

TSP (Team Software Process)：使用脚本启动每个项目，团队自指导，鼓励度量并以改进团队过程为目标分析度量。
