2.1 软件工程的定义与层次

本章核心问题：什么是软件工程——它不仅是写代码，而是一套系统化、规范化的工程方法体系。

IEEE 与 NATO 定义 必考

IEEE 定义（两个层面）：

  1. 将系统化的、规范的、可量化的方法应用于软件的开发、运行和维护，即将工程化方法应用于软件
  2. 对上述方法的研究

NATO 1968 定义：软件工程是建立并运用可靠的工程化原则，以经济的方式开发在真实机器上可靠高效运行的软件。

核心公式：软件工程 = 工程化原则 + 软件开发

易错：IEEE 定义有两层含义——(1) 应用工程化方法 (2) 研究方法本身。考试常只考第一层而忽略第二层。NATO 定义强调「经济性」和「可靠性」，这是区分关键词。

软件工程的层次化技术 重点

软件工程是分层技术，自下而上为：

层次名称作用
底层基石质量关注 (Quality)整个软件工程的根基，一切活动的出发点和目标
第二层过程模型 (Process)将开发活动有效组织起来，是项目管理和控制的基础，定义交付物、里程碑等
第三层方法 (Methods)从技术上告诉我们怎么做——沟通、需求分析、设计、构造等
最上层工具 (Tools)支撑软件工程方法的自动化或半自动化工具，如 CASE 工具
易错：质量是「根基」不是「目标之一」——没有质量关注，过程和方法都没有意义。常见判断题：说「过程是软件工程的基础」是对的，但这基础建立在质量之上。

2.2 软件过程

核心问题：如何把软件开发中的各种活动系统化地组织起来，形成可管理、可重复的过程。

过程、活动、动作、任务 重点

层级定义示例
过程 (Process)为创建某个工作产品而执行的活动、动作、任务的集合整个软件开发过程
活动 (Activity)追求广泛目标，与领域、项目规模、复杂度、严格度无关与利益相关者沟通
动作 (Action)包含一组产生主要工作产品的任务架构设计（产出架构模型）
任务 (Task)聚焦小而明确的目标，产出具体成果执行单元测试
易错：活动 > 动作 > 任务，三者粒度递减。活动关注「做什么」，动作关注「产出什么」，任务关注「具体怎么做」。常见混淆：把架构设计说成「活动」而非「动作」。

通用过程框架 必考

过程框架 (Process Framework)：通过识别少量适用于所有软件项目的框架活动，为完整的软件过程建立基础——无论项目规模或复杂度。

五大框架活动：

活动核心内容
沟通 (Communication)客户协作与需求收集
计划 (Planning)建立工程工作计划，描述技术风险，列出资源需求、工作产品，定义进度
建模 (Modeling)创建模型帮助开发者和客户理解需求和软件设计
构建 (Construction)代码生成与测试
部署 (Deployment)交付软件供客户评估和反馈

普适性活动 (Umbrella Activities) 重点

普适性活动贯穿整个过程，与框架活动互补：

活动职责
软件项目跟踪与控制评估进度，必要时采取纠正措施保持进度
风险管理评估可能影响项目结果或产品质量的风险
软件质量保证 (SQA)定义并执行确保软件质量所需的活动
技术评审评估工作产品，在上游发现并消除错误
度量定义和收集过程、项目、产品度量数据
软件配置管理 (SCM)管理整个过程中变更带来的影响
复用管理定义工作产品复用标准，建立复用机制
工作产品准备与生产创建模型、文档、日志、表单、列表等
易错：普适性活动不等同于框架活动。框架活动是「阶段」（沟通→计划→建模→构建→部署），普适性活动是贯穿始终的「保护伞」。

过程适应性调整 高频

过程不是教条，可根据项目特点、团队特性、组织文化进行调整。调整维度包括：

  • 活动/动作/任务的总体流程和相互依赖关系
  • 框架活动中动作和任务细化的程度
  • 工作产品的定义和要求程度
  • 质量保证活动应用的方式
  • 项目跟踪和控制活动应用的方式
  • 过程描述的详细程度和严谨程度
  • 客户和利益相关者参与程度
  • 软件团队的自主权
  • 团队组织和角色的明确程度

2.3 软件工程实践

核心问题：如何将软件工程理论落地为日常实践。

实践的精髓 重点

步骤内容对应过程框架
1. 理解问题沟通与分析沟通 + 计划
2. 策划方案建模与软件设计（写代码前先设计！）建模
3. 执行计划代码生成构建
4. 检查结果测试与质量保证构建 + 部署

通用原则 高频

原则含义
存在价值只为用户提供价值而存在
KISS — 保持简洁不要上来就复杂化
保持愿景在取舍中不忘记最初的目标
关注使用者从使用者角度思考
面向未来长远考虑，不过于短视
提前计划复用不要等规模大了才考虑复用
认真思考通盘考虑后再行动
易错：KISS 原则不是「降低功能复杂度」，而是「不引入不必要的复杂度」。保持愿景不是说需求不变更，而是在不可避免的取舍中不迷失方向。

2.4 软件开发神话

核心问题：破除软件开发中常见的错误观念，建立正确的工程思维。

管理神话 重点

神话事实
有一本标准规程手册就够了没人真正关心和执行，手册不等于实践
进度落后就加人赶工人月神话：向滞后的项目添加人手只会使其更滞后（1+1 << 2）——新人需要老人培训，反而拖慢进度
外包出去就可以放手不管如果管不好自己的团队，外包也必然失败
易错：「人月神话」是考试高频考点。核心论证：软件开发不是机械制造，新人加入需要沟通成本和培训时间，边际效益递减甚至为负。

客户神话 重点

神话事实
有大概目标就可以开始写代码，细节后面补充「完成 90%」陷阱——开发者在剩余时间里始终「90% 完成」。不明确的需求导致无限返工
软件是弹性的，需求变更容易适应变更代价随时间指数增长：早期变更成本低，后期变更可能推翻架构，代价极高

从业者神话 必考

神话事实
程序写完交付就完事了60%~80% 的工作量发生在首次交付之后（维护、修复、扩展）
程序跑起来之前无法评估质量技术评审 (Technical Review) 是比测试更有效的质量过滤器，从项目启动即可实施
可执行程序是唯一可交付成果软件配置 = 程序 + 文档 + 数据。文档是成功开发和维护的基础
软件工程产生大量无用文档，拖慢效率软件工程的目标是质量，不是文档。好的质量 → 减少返工 → 更快的交付速度
易错：判断——「软件工程会降低开发效率」是错误的。实际上：软件工程 → 提高质量 → 减少返工 → 交付更快。文档是手段，质量是目的。
