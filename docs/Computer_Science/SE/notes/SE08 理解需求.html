需求工程七步法 必考

核心问题：构建软件系统最困难的部分是决定要构建什么——三大问题：范围问题、理解问题、易变性问题。

1. 起始 (Inception)

提出上下文无关问题建立基本理解：问题的基本理解 → 谁想要解决方案 → 期望的解决方案本质 → 前期沟通协作的有效性。识别利益相关者，识别多种视角，向协作努力。

2. 获取 (Elicitation) 重点

从所有利益相关者获取需求。会议由软件工程师和客户共同参与，制定准备和参与规则，由引导者控制会议。使用定义机制（工作表/白板/电子公告板）。目标：识别问题 → 提出方案元素 → 协商不同方法 → 制定初步需求集。

3. 细化 (Elaboration)

创建分析模型，识别数据、功能和行为需求。

4. 协商 (Negotiation) 重点

识别关键利益相关者 → 确定每个利益相关者的「获胜条件」 → 协商 → 达成「双赢」需求集。如果不同客户/用户无法就需求达成一致，失败风险非常高。

5. 规格说明 (Specification)

可以是书面文档、模型集、形式化数学描述、用户场景集（用例）、原型中的任意一种或多种。

6. 验证 (Validation) 必考

审查需求中的：内容或解释错误、需要澄清的地方、缺失信息、不一致（大型系统的主要问题）、冲突或不现实的需求。验证问题清单：是否与总体目标一致？抽象层次是否合适？是否必要？是否无歧义？是否有归属？是否冲突？是否可实现？是否可测试？

7. 需求管理

持续跟踪、监控和调整需求，尤其在增量开发中——分布式调试、运行时验证、运行时确认、业务活动监控、演化与协同设计。

质量功能部署 (QFD) 重点

三类需求：正常需求 (Normal)——明确陈述的；期望需求 (Expected)——隐含的、基本的；兴奋需求 (Exciting)——超出期望的增值特性。

非功能需求 (NFR) 高频

质量属性、性能属性、安全属性或系统约束。两阶段分析：创建 NFR× 指南矩阵 → 团队对每对 NFR/指南分类（互补/重叠/冲突/独立）并排优先级。

获取工作产品

需求与可行性声明、系统和产品范围声明、利益相关者列表、技术环境描述、需求列表与领域约束、使用场景集、任何为更好定义需求而开发的原型。

用例 (Use-Cases) 必考

从参与者 (Actor)视角描述使用场景的线程。每个场景回答：主要/次要参与者是谁？参与者目标是什么？前置条件？主要任务/功能？可能的扩展和变化？参与者获取/产生/修改什么信息？

分析模型元素

  • 场景元素：用例、活动图
  • 类元素：类图
  • 行为元素：状态图
  • 流元素：数据流图

分析模式

模式名 → 意图 → 动机 → 力量与上下文 → 解决方案 → 后果 → 设计 → 已知使用 → 相关模式。
