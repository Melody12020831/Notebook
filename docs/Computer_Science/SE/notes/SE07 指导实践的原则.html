过程原则 (Principles that Guide Process)

核心观点：Steve McConnell 提出软件工程原则没有 3 年半衰期——它们将服务整个职业生涯。

过程原则 I 重点

  1. 保持敏捷：无论选择规约式还是敏捷过程模型，敏捷开发的基本原则应指导方法
  2. 每步关注质量：每个过程活动/动作/任务的出口条件应聚焦于产生的工作产品质量
  3. 准备适应：过程不是宗教体验，必要时根据问题、人员和项目约束调整方法
  4. 建立有效团队：过程和实最重要，但底线是人——建立互信互尊的自组织团队

过程原则 II

  1. 建立沟通和协调机制：项目失败往往因为重要信息遗漏和/或利益相关者协调失败
  2. 管理变更：建立正式或非正式机制管理变更的请求、评估、批准和实施
  3. 评估风险：制定应急计划，很多事都可能出错
  4. 创造对他人有价值的工作产品：只为其他过程活动/动作/任务提供价值的产品才值得创建

实践原则 (Principles that Guide Practice) 必考

实践原则 I

  1. 分而治之：分析和设计应始终强调关注点分离 (Separation of Concerns, SoC)
  2. 理解抽象的使用：抽象是对系统复杂元素的简化，用单个短语传达含义
  3. 力求一致性：熟悉的上下文使软件更易使用
  4. 关注信息传递：特别关注接口的分析、设计、构建和测试

实践原则 II

  1. 构建有效模块化的软件：关注点分离是哲学，模块化是实现哲学的机制
  2. 寻找模式：模式的目标是创建知识体系帮助开发者解决重复出现的问题
  3. 从多个视角表示问题和解决方案
  4. 记住有人将维护软件
易错：SoC（关注点分离）是原则，模块化是实现手段。两者密切相关但概念不同——判断常混为一谈。

沟通原则 重点

  1. 倾听——专注于说话者的内容，而非准备回应
  2. 沟通前准备——先花时间理解问题
  3. 有人应引导活动——每次沟通会议应有引导者
  4. 面对面沟通最佳——但有相关信息的其他表示时效果更好
  5. 做笔记和记录决策
  6. 力求协作——集体知识融合产生共识
  7. 保持聚焦，模块化讨论
  8. 不清楚就画图
  9. (a) 同意就进入下一步 (b) 不同意也前进 (c) 不清楚且无法澄清也前进
  10. 谈判不是竞赛——双赢才是最好结果

计划原则

  1. 理解项目范围——范围提供目的地
  2. 让客户参与计划活动
  3. 计划是迭代的——从不是刻在石头上
  4. 基于已知进行估算
  5. 定义计划时考虑风险
  6. 实事求是——人不是每天 100% 效率
  7. 随计划展开调整粒度
  8. 定义如何确保质量
  9. 描述如何适应变更
  10. 频繁跟踪计划并做调整——项目落后是一天天累积的

建模原则 必考

两类模型：需求模型（分析模型）——从信息域/功能域/行为域描述客户需求；设计模型——描述帮助构建软件的特征（架构/UI/组件细节）。

敏捷建模原则

首要目标构建软件而非创建模型；轻装上阵——不要创建超过需要的模型；创建最简单的模型；模型应易于变更；每个模型有明确目的；根据系统适配模型；追求有用而非完美；不要成为模型语法的教条者。

需求建模原则

信息域必须被表示和理解；软件功能必须被定义；软件行为必须被表示；模型必须分层揭示细节；从基本信息向实现细节推进。

设计建模原则

设计应可追溯到需求模型；始终考虑架构；数据设计与处理功能设计同等重要；接口需精心设计；UI 设计适应最终用户需求；组件级设计应功能独立；组件应松散耦合；设计模型应易理解；设计应迭代开发。

Living Modeling 原则

以利益相关者为中心的模型；模型与代码紧密耦合；模型与代码间双向信息流；创建公共系统视图；模型信息持久化以跟踪变更；跨模型层信息一致性验证；每个模型元素有指定权限和责任。

构建原则 重点

准备原则

写代码前：理解问题、理解设计原则、选择合适语言、选择编程环境、创建一组单元测试。

编码原则

结构化编程、考虑结对编程、选择合适的数据结构、理解架构一致接口、简单条件逻辑、嵌套循环易测试、有意义变量名、自文档化代码、可视化布局。

验证原则

代码走查 (Code Walkthrough) → 执行单元测试并修正错误 → 重构代码 (Refactor)。

测试原则 重点

  1. 所有测试应可追溯到客户需求
  2. 测试应在测试开始前很久就计划好
  3. 帕累托原则适用于软件测试（80% 错误来自 20% 组件）
  4. 测试应从小开始向大推进
  5. 穷尽测试不可能
  6. 测试工作量与预期故障密度匹配
  7. 静态测试可产生高回报
  8. 跟踪缺陷并在缺陷中寻找模式
  9. 包含展示软件正确行为的测试用例

部署原则

  1. 管理客户对软件的期望
  2. 组装和测试完整的交付包
  3. 在软件交付前建立支持机制
  4. 向最终用户提供适当的指导材料
  5. 有缺陷的软件应先修复，后交付
