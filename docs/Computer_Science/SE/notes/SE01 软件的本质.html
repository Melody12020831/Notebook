1.1 软件的本质

核心问题：软件到底是什么——它不只是代码，而是一种被开发而非制造的逻辑产品。

软件的定义 必考

软件是一组构成配置 (Configuration) 的要素或对象的集合，包括：

  1. 指令（计算机程序）：执行时提供所需功能和性能
  2. 数据结构：使程序能充分处理信息
  3. 文档：描述程序的操作和使用

关键区别：软件是被开发/工程化 (developed/engineered) 的，而非传统意义上的制造 (manufactured)。

易错：软件与硬件的本质区别——软件是逻辑产物（开发），硬件是物理实体（制造）。软件不会「磨损」但会「退化」。

软件退化曲线 重点

硬件失效曲线是「浴缸曲线」：初期失效率高 → 平稳期 → 磨损期失效率升高。

软件不会磨损，但会退化 (deteriorate)：每次修改引入新错误，失效率突然升高。没有软件备件。

易错：判断题——「软件会像硬件一样磨损」是错误的。软件不会被磨损，但维护过程中的修改会导致退化。

软件的双重角色

软件 = 产品（信息转换器）+ 产品交付载体（OS/网络/工具）

遗留软件必须变更的原因 高频

  1. 适应：适应新计算环境或技术
  2. 增强：实现新业务需求
  3. 扩展：与其他现代系统或数据库互操作
  4. 重新架构：使其在网络环境中可行

软件危机的历史 重点

IBM 360 操作系统（1963-1966）：5000 人-年工作量，最多 1000 人同时开发，近 100 万行源码。每次新版本从前版本修复约 1000 个错误。

Brooks 总结：「程序设计工作正像一个泥潭…一批批程序员被迫在泥沼中拼命挣扎。」——由此诞生经典著作《人月神话》。

易错：IBM 360 案例是「软件危机」最著名的例子，Brooks 从中总结出人月神话的核心观点。

软件开发的根本问题

  • 为什么软件需要这么长时间才能完成？
  • 为什么开发成本如此之高？
  • 为什么在交付前无法找到所有错误？
  • 为什么花这么多时间维护已有程序？
  • 为什么难以度量开发过程中的进展？

1.2 软件的变化本质

核心问题：软件形态随时代不断演变——从单机程序到网络应用、移动应用、云端服务、AI 驱动软件。

WebApps 高频

现代 WebApps 不仅仅是超文本+图片。包含 XML/Java 增强交互性，可与数据库和业务应用集成。Web 3.0 / 语义 Web：语义数据库、灵活数据表示、API 访问。内容的美学品质是质量的重要决定因素。

移动应用 高频

驻留在移动平台，UI 考虑设备特性和位置属性。提供持久化存储。移动 Web App（浏览器访问）vs 移动 App（直接访问硬件，本地处理和存储）。

云计算 重点

三层模型：IaaS（基础设施即服务）→ PaaS（平台即服务）→ SaaS（软件即服务）

云架构包含前端服务（客户端设备+应用软件）和后端服务（服务器+数据存储+驻留应用），可分段限制私有数据访问。

产品线软件 高频

产品线软件：共享一组共同特性、满足特定市场需求的一组软件密集型系统。利用公共核心可复用组件，共享需求、架构、设计模式、测试用例等资产。

Software 1.0 / 2.0 / 3.0 必考

版本核心编程方式代表
Software 1.0计算机代码程序员编写显式指令传统程序、GitHub 代码库
Software 2.0神经网络权重数据训练模型参数AlexNet (2012)、HuggingFace 模型
Software 3.0LLM 提示词自然语言提示GPT 系列、可编程神经网络 (2019)

Vibe Coding（氛围编程，2025）：Andrej Karpathy 提出，通过自然语言提示与 AI 生成代码，强调开发者进入「创意 flow」状态。

易错：三个版本不是替代关系。Software 2.0 不是取代 1.0，而是增加了新的编程范式。Software 3.0 进一步用自然语言提示取代部分代码编写。
