传统软件测试生命周期回顾

核心问题：LLM 能显著提升测试效率和覆盖度，但也引入LLM 自身的安全风险——测试和安全是硬币的两面。

LLM 驱动测试 必考

  • 测试用例自动生成：从需求规格/代码/API 描述自动生成测试用例
  • 测试数据生成：自动构造边界值、异常输入、组合爆炸数据
  • 测试脚本自动编写：生成 JUnit/Pytest 等测试框架代码
  • 缺陷定位：分析日志和错误报告定位可能的根因

LLM 驱动安全 必考

  • 漏洞自动检测：扫描代码识别 SQL 注入、XSS、CSRF 等常见漏洞
  • 攻击面分析：从系统描述自动识别潜在攻击入口
  • 安全测试用例生成：自动构造渗透测试场景

LLM 自身的安全风险 重点

  • 提示注入 (Prompt Injection)：攻击者在输入中嵌入恶意指令劫持 LLM 行为
  • 数据泄露：LLM 可能在输出中泄露训练数据或用户输入的敏感信息
  • 幻觉导致的误报/漏报：LLM 可能漏报真实漏洞或对安全代码误报
  • 对抗攻击：精心设计的输入使 LLM 产生错误的安全分析结果
易错：LLM 既是安全工具（发现漏洞）也是攻击面（本身可能有漏洞）。用 LLM 做安全测试本身需要安全保障——这就是「测试测试者」的元问题。
