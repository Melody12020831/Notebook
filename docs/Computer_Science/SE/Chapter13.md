# Chapter 13 | Architectural Design

## 架构的核心定义与工程价值

1. 架构的本质

- 架构绝不是最终可运行的软件代码，而是一种高度抽象的设计表示形式（蓝图）。

2. 为什么需要架构 (Why Architecture)

- 需求验证：用于分析和推演系统设计在满足既定需求时的有效性。
- 决策缓冲：在项目早期（修改成本极低时），提供评估和对比不同架构替代方案的可能。
- 风险控制：从顶层大幅降低后续代码构建过程中的技术与集成风险。

3. 架构的重要作用 (Why is Architecture Important)

- 沟通枢纽：它是促进所有系统利益相关者（Stakeholders，如客户、研发、运维等）进行有效沟通的核心媒介。

4. 架构决策的深远影响

- 架构代表了系统最核心的“早期设计决策”，这些决策具有极强的锁定效应，直接决定后续工程的走向及系统的最终成败。

5. 智力可控性 (Intellectually graspable)

- 架构是对复杂系统的极简抽象，它过滤了底层代码细节，为工程师提供了一个人类智力可以轻松理解和把握的宏观协作模型。

6. 行业标准支持

- IEEE计算机协会专门为架构描述制定了规范：IEEE-Std-1471-2000《软件密集型系统架构描述的推荐实践》。

---

### IEEE 架构描述标准 (IEEE-Std-1471-2000)

1. 标准的核心目的

- 统一概念框架与词汇表，消除行业沟通壁垒。
- 提供规范的架构描述表达指南。
- 鼓励并推广稳健的架构设计最佳实践。

2. 架构描述 (Architectural Description, AD)

- 定义：用于全面记录和展示一个架构的各种文档和产物的总和（集合）。

3. 多重视图 (Multiple views) 原则

- 核心理念：架构描述绝不能是单一维度的，必须采用多重视图。
- 原因：系统拥有众多角色不同的利益相关者（Stakeholders），每个视图需要专门针对某一组特定的“关注点”，从该视角的维度去映射并表示整个系统。

---

### 架构的分类学体系

1. 架构体裁/流派 (Architectural Genres)

- 定义：指代整个庞大软件领域中的某一个特定的宏观业务与技术类别（如：人工智能、实时交易、嵌入式等）。

2. 架构的层级细分 (以建筑学作类比)

- 宏观体裁 (Genre)：类似于建筑领域的顶层分类。
- 子类别 (Subcategories)：在体裁内部继续细分的领域。
- 架构风格 (Styles)：如办公楼、公寓等具体形式。
- 架构模式 (Patterns)：在风格内部使用的、可预测的具体结构化方案。
- 意义：帮助设计师迅速定位问题域，匹配行业内已被验证的成熟结构。

3. 架构风格 (Architectural Styles) 核心定义

- 架构风格描述了特定系统结构的基本范式。任何一种标准的架构风格，都必然包含以下四大基石要素：
a. 组件 (Components)：

- 架构的执行实体（如数据库、具体计算模块等），负责完成系统要求的功能。

b. 连接器 (Connectors)：

- 架构的通信桥梁，负责实现各组件之间的通信、协调与交互合作机制。

c. 约束 (Constraints)：

- 架构的集成纪律，严格定义各类组件可以怎样被组装和连接（限制非法交互）。

d. 语义模型 (Semantic models)：

- 架构的分析与推演工具，使设计师可以通过评估已知的局部组件特性，从整体上理解和推演系统的全局属性。

## 经典软件架构风格 (Architectural Styles) 

业界最核心的五大经典架构风格流派：

- 以数据为中心的架构 (Data-centered architectures)
- 数据流架构 (Data flow architectures)
- 调用和返回架构 (Call and return architectures)
- 面向对象架构 (Object-oriented architectures)
- 分层架构 (Layered architectures)

---

### 以数据为中心的架构 (Data-Centered Architecture)

- 核心结构：系统的物理或逻辑中心是一个全局的“数据存储 (Data store)”，如关系型数据库、数据仓库或黑板系统。
- 交互模式：所有的“客户端软件 (Client Software)”呈星型拓扑围绕中心分布，只与数据中心进行读/写交互，客户端之间不发生直接通信。
- 核心优势：

    - 极高的数据共享能力与一致性。
    - 极强的可扩展性（客户端解耦）：添加、修改或删除某一个客户端软件，完全不会影响其他客户端的运行，系统扩展极其方便。

---

### 数据流架构 (Data Flow Architecture)

- (概念引入：侧重于刻画数据在系统内部像流水一样被持续转换和处理的动态过程)
- 数据流架构的核心模式：管道-过滤器 (Pipe-and-Filter)

- 两大核心要素：
    - 过滤器 (Filter)：独立的处理组件。其唯一职责是读取输入数据流，进行特定的内部处理/转换，然后将结果产生为输出数据流。过滤器之间彼此盲目（不清楚上下文）。
    - 管道 (Pipes)：负责在过滤器之间传输和缓冲数据的连接器。

- 结构特征：数据输入后，通过一系列串行或并行的过滤器加工，最终输出。
- 核心优势：

    - 极致的重用性与可重构性：过滤器可以像乐高积木一样被任意组合、替换和复用。
    - 并发潜力：不同的过滤器可以同时处理数据流的不同部分。

---

### 调用和返回架构 (Call and Return Architecture)

- (概念引入：基于传统的结构化程序设计思想，强调控制权的自顶向下转移与结果返回)
- 调用和返回架构的核心特征 (主程序/子程序架构)

    - 运行机制：控制权自顶向下流转。主程序调用子模块，子模块完成计算后将控制权及结果返回给上级。
    - 形态特征：呈现出典型的树状层级控制结构。

- 评估该架构复杂度的四大核心结构化指标：

    - 深度 (Depth)：调用树的垂直嵌套层级数。
    - 宽度 (Width)：系统在同一控制层级上模块分布的最大水平跨度。
    - 扇出 (Fan-out)：某一个特定模块直接控制/调用的下级模块数量。(建议保持在合理范围内，扇出过大代表该模块控制逻辑过载)。
    - 扇入 (Fan-in)：某一个特定模块被多少个上级模块所调用。(高扇入是优秀设计的标志，代表该模块具有极高的复用价值)。

---

### 面向对象架构 (Object-Oriented Architecture)

- 核心构建块：对象 (Objects) 与类 (Classes)。
- 运行机制：将数据及操作数据的算法严密封装在对象内部，对象与对象之间通过发送“消息”进行交互和功能调用。
- 核心优势：

    - 高度封装，符合信息隐藏原则，安全性高。
    - 借助继承与多态机制，极大地提升了系统的可复用性和弹性扩展能力。

- 面向对象架构的工业级实现代表：J2EE

  - J2EE (Enterprise Java Software) 是一套成熟的企业级面向对象架构规范与组件模型。
  - 意义：为开发者提供了标准的、基于对象的分布式组件运行环境，避免了底层通信与事务处理的“重复造轮子”。

- 面向对象架构的标准制定者：ECMA International

  - 作用：制定并维护了众多核心的面向对象语言与基础结构标准 (如 ECMAScript/JavaScript, C#, CLI)。
  - 启示：优秀的架构设计应依托并遵循国际通用标准，以确保软件生态的兼容性与互操作性。

---

### 分层架构 (Layered Architecture)

- 业务背景：面对包含如任务管理、工作流实例、业务规则调度等极其复杂的综合性应用时，单体结构会陷入混乱。
- 解决思路：必须引入分层思想，将不同粒度和职责的模块剥离到不同的层次中去。
- 分层架构的核心结构与约束 (洋葱模型)

    - 典型层级划分 (自外向内)：

        1. 用户界面层 (User interface layer)：管理人机交互。
        2. 应用层 (Application layer)：处理具体的业务逻辑。
        3. 实用工具层 (Utility layer)：提供公共支撑组件库。
        4. 核心层 (Core layer)：底层的操作系统接口或基础核心数据服务。

    - 核心架构约束 (纪律)：

        - 单向依赖：外层服务调用内层服务，内层对整个外层环境一无所知，且不能反向调用。

    - 核心优势：高内聚、低耦合，某一层级的内部重构或替换不会对其他层级产生破坏性波及。

- 企业级分层架构全景实例解析

标准企业级系统自顶向下的典型层级拓扑：

1. 多端接入层：浏览器、移动端PDA、传统客户端等。
2. 应用业务层：按具体业务域划分的组件集合 (如文档管理、库存管理)。
3. 关键共性服务层：跨业务复用的高级服务引擎 (如工作流引擎、报表引擎)。
4. 通用基础服务层：系统级支撑能力 (如安全权限管控、事务控制、日志记录)。
5. 数据访问引擎层：屏蔽底层数据库差异的统一网关/ORM层。
6. 物理存储层：真实的关系型数据库或分布式文件系统。

架构启示：通过极其细腻的层次切分，完美贯彻了“关注点分离”理念。每一层只专注自己的核心职责，层与层之间仅通过标准接口通信。

## 架构模式 (Architectural Patterns)

1. 架构模式的概念

- 针对架构设计中特定领域复杂问题的标准化、可复用的解决方案。

2. 核心架构模式分类

- 并发模式 (Concurrency)：

    - 目标：使应用程序能够模拟并行，高效处理多个同时发生的任务。
    - 常见实现：操作系统进程管理模式、任务调度器模式。

- 持久化模式 (Persistence)：

    - 目标：确保数据的生命周期超越创建该数据的进程 (即断电/关闭程序后数据不丢失)。
    - 常见实现：

        1. DBMS模式：直接利用外部数据库管理系统的存储与检索能力。
        2. 应用级持久化模式：在应用程序架构内部封装持久化特性。

3. 分布式模式 (Distribution)：

   - 目标：解决分布式环境下系统或组件间的跨网络通信问题。
   - 常见实现：代理模式 (Broker)，在客户端与服务端之间引入“中间人”进行解耦和通信路由。

## 架构设计的推演步骤 (Architectural Design Process)

1. 第一步：定义架构上下文 (Architectural Context)

- 核心任务：将待开发的软件置于真实环境中，明确系统的边界。
- 关键动作：定义与该软件发生交互的所有“外部实体” (如外围系统、硬件设备、最终用户)，并明确交互的数据流向与性质。

2. 第二步：识别架构原型 (Architectural Archetypes)

- 核心任务：提炼代表系统核心运行机制的基础元素。
- 原型定义：一种高度的抽象 (类似于面向对象中的抽象类)，它代表了系统某一部分最本质的行为特征。

3. 第三步：细化并指定系统结构 (Specify the structure)

- 核心任务：将高度抽象的“架构原型”向下推演。
- 关键动作：设计人员通过定义、拆解和细化具体的“软件组件”来实现每一个原型，从而最终敲定整个架构的内部物理/逻辑结构。
- 架构上下文图 (Architectural Context Diagram) 实例解析
    - 视角：将目标系统视为“黑盒”，不暴露任何内部组件细节。
    - 核心要素：

        1. 目标系统 (Target System)：位于图中央，代表待设计的软件边界。
        2. 外部实体 (External Entities)：环绕在系统四周的参与者 (如用户、硬件传感器、第三方网络系统等)。
        3. 交互通道：连接外部实体与目标系统的带方向箭头，代表数据流或控制流 (如 Uses 依赖关系)。

    - 工程意义：强制架构师在深入细节前，先全盘理清系统的系统边界、外部依赖以及所有必要的通信接口。

- 架构原型 (Archetypes) 实例解析

- 核心作用：滤除庞杂的业务细节，抽象出系统最底层的运作骨架与核心逻辑。
- 典型原型提取 (以安防系统为例)：

1. 控制器 (Controller)：代表中枢调度与管理机制。
2. 节点 (Node)：代表基础的可接入端点，被控制器统一管理 (聚合关系)。
3. 探测器 (Detector)：作为节点的派生，抽象了所有负责信息输入的“感知类”行为。
4. 指示器 (Indicator)：作为节点的派生，抽象了所有负责信息输出与动作执行的“反馈类”行为。

- 意义：为架构提供了极其稳固的扩展基石。未来任何新增的业务组件，都可以向上映射到这些原型之中。

- 初始组件结构图 (Initial Component Structure) 实例解析

  - 顶层调度：通常存在一个核心的执行/调度组件 (如 SafeHome Executive)，负责整体系统级的功能路由与控制流分发。
  - 业务子系统划分：将系统划分为高内聚的粗粒度业务组件 (如安全子系统、监控子系统、外部通信子系统)。
  - 初步拆解：对粗粒度组件进行第一层的职责拆分，形成基础的模块树。

4. 第四步：细化组件结构 (Refined Component Structure)

- 核心逻辑：应用“逐步求精”的设计原则，对初始的粗粒度组件进行自顶向下的深度拆解。
- 细化组件结构图 (Refined Component Structure) 实例解析

- 细化到底层物理/控制接口：将中等粒度组件彻底拆解为底层执行单元。
    - 例如，将“面板处理”拆分为具体的“键盘捕获”与“屏幕显示”控制组件。
    - 将“警报处理”拆分为“电话通信调度”与“物理警报触发”组件。
- 工程意义：此时的组件树已经细化至“叶子节点”，彻底消除了架构设计中的模糊地带，开发人员可以直接基于这些叶子节点展开组件级算法设计与代码编写。

## 优秀架构的五大核心考量维度 (Architectural Considerations)

评估一个软件架构是否卓越，应从以下五个维度进行检验：

1. 经济性 (Economy)：追求极致的整洁与极简。通过高度的抽象机制，剔除所有不必要的架构细节与冗余。
2. 可见性 (Visibility)：强调架构的“可解释性”。所有的架构设计决策及其背后的权衡原因，必须对后续接手和审查模型的工程师透明且显而易见。
3. 间距 / 分离度 (Spacing)：严格落实“关注点分离”。确保在拆分模块时，组件之间绝对没有暗藏的“隐藏依赖 (Hidden dependencies)”。
4. 对称性 (Symmetry)：强调系统整体的一致性与平衡感。系统各部分的架构风格、接口规范和属性分布应当协调统一。
5. 涌现性 (Emergence)：架构的高级形态。系统各组件在协同运行时，能够产生一种适应环境变化的、自组织的动态行为与控制能力。

---

### 架构权衡分析 (Architectural Tradeoff Analysis, ATAM)

1. **核心目的**：系统化地评估软件架构，明确质量属性（如性能、安全性）之间的冲突，并做出合理的设计权衡。
2. **ATAM 评估的 6 个标准步骤**：

- 步骤 1：收集业务场景。
- 步骤 2：引出并明确需求、约束及运行环境描述。
- 步骤 3：使用多重视图（模块、过程、数据流）详细描述候选的架构风格与模式。
- 步骤 4：孤立地评估各项系统的“质量属性”。
- 步骤 5：进行 **敏感性分析**（找出特定架构属性变化时，对整体质量属性影响最大的敏感点）。
- 步骤 6：基于敏感性分析的结果，对候选架构进行客观评估与批判。

---

### 架构描述与审查 (ADL & Architecture Reviews)

1. **架构描述语言 (ADL)**

- **定义**：一种提供严格语义和语法来形式化描述软件架构的语言。
- **核心能力**：允许设计人员精确表达组件的分解（向下拆分）、组合（向上封装）以及组件间接口交互的连接机制。

2. **架构审查 (Architecture Reviews)**

- **核心目的**：评估架构方案对质量需求的满足度，并提前识别系统底层的潜在风险。
- **工程价值**：“防患于未然”。通过在早期（编码前）发现底层设计缺陷，大幅度降低项目返工与推进的成本。

3. **常见的审查手段**

- 基于经验的专家审查
- 技术原型 (Prototype) 评估
- 业务场景推演
- 规范检查表 (Checklists)

4. **基于模式的架构审查流程 (Pattern-Based Review)**

- 步骤 1/2：结合业务用例与需求，审视当前的架构图纸。
- 步骤 3：识别设计中使用的具体 **架构模式**，检查其匹配度。
- 步骤 4：通过文档与推演，评估采用该模式后对系统质量属性（如性能、弹性）产生的实际影响。
- 步骤 5：暴露并记录下引入该模式后产生的所有质量痛点与隐患。

## 敏捷开发中的架构 (Agility and Architecture)

1. **核心理念冲突与融合**

- 敏捷强调快速响应与编码，但完全摒弃架构会导致灾难性的后期返工。因此敏捷同样需要架构。

2. **“行走的骨架” (Walking skeleton)**

- 在大规模编码启动前，基于当前的用户故事，搭建并演进出的一个最小化、能跑通整体链路的基础架构模型。

3. **敏捷中的架构演进机制**

- **故事板驱动**：架构师也可以像产品经理一样，将底层的技术架构需求写成“用户故事”，融入敏捷迭代板中。
- **持续审查**：敏捷项目以冲刺 (Sprint) 为周期。在敏捷中，直接对每个 Sprint 产出的核心层代码进行 Review (代码审查)，这就是最实用、最高频的架构审查形式。
