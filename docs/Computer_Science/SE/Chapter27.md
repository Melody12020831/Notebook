# Chapter 27 | Security Engineering

## Why Security Engineering?

安全并非系统的“附属品”，而是其核心组成部分。

* **安全的基本定位：** 安全是保证系统**完整性**（Integrity）、**可用性**（Availability）、**可靠性**（Reliability）和**安全性**（Safety）的先决条件。
* **安全的作用：** 它提供了相应的机制，使得系统能够保护其**资产**（Assets）免受攻击。
* **什么是资产？** 资产是指对利益相关者（Stakeholders）有价值的任何系统资源，包括：信息、文件、程序、存储空间和处理器能力。
* **什么是攻击？** 攻击利用系统中的漏洞，从而获取未经授权的系统访问权限。
* **核心结论：** 仅靠“修补漏洞”（响应漏洞报告）是很难提高系统安全性的。**安全必须在设计之初就融入系统（Designed in from the beginning）**，并在整个过程中进行适当的测试。

## 分析安全需求 (Analyzing Security Requirements)

在项目初期如何量化和分析安全需求。

* **暴露度（Exposure）：** 这是一个量化指标。它指代为了重新创建一个丢失的系统资产，所需要花费的**时间或成本**。这个数值可以帮助开发者判断某个资产的重要程度。
* **威胁分析（Threat Analysis）：** 这是一个识别过程。即确定哪些具体情况或威胁可能损坏系统资源，或者导致资源无法被正常访问。
* **控制（Controls）：** 这是应对措施。即通过设计特定的机制（控制手段）来预防攻击，并减轻在遭受攻击时系统可能产生的损害。

## 在线安全威胁 (Online Security Threats)

当前互联网时代下，四个最主要的在线安全挑战：

* **社交媒体（Social Media）：** 社交网络平台经常允许第三方开发应用，这些应用往往能够轻易获取用户的个人详细隐私信息。
* **移动应用程序（Mobile Applications）：** 运行在手机等移动设备上的原生 App，通常拥有与设备所有者相同的系统权限，这带来了潜在的权限滥用风险。
* **云计算（Cloud Computing）：** 云服务模糊了“可信内部（trusted inside）”和“不可信外部（untrusted outside）”之间的界限，这引入了额外的机密性和信任问题。
* **物联网（Internet of Things）：** 日常生活中的各种设备（如智能家电、传感器）都具备了通信能力，它们会不断报告关于用户及其所处环境的上下文信息，这使得隐私泄露的场景更加广泛。

## 安全工程分析 (Security Engineering Analysis)

将安全理念转化为具体工程实践的四个关键步骤：

* **安全需求获取 (Security requirements elicitation)：**
    * 明确用户如何与系统资源交互。
    * **编写“滥用故事 (Abuser stories)”：** 从攻击者角度描述系统可能受到的威胁。
    * 利用威胁建模和风险分析来制定非功能性需求中的安全策略。
    * 通过攻击模式识别来寻找解决系统缺陷的方案。
* **安全建模 (Security modeling)：**
    * 涵盖政策目标、接口需求、安全架构等。
    * **状态模型 (State models)：** 确保系统在整个生命周期内，其状态转换始终处于“安全状态”。    * 使用形式化模型（Formal models）进行正确性证明，提升系统可信度。
* **度量设计 (Measures design)：** 安全指标应聚焦于系统的**可靠性、可信度和生存能力**（dependability, trustworthiness, and survivability）。
* **正确性检查 (Correctness checks)：** 确保所有的测试用例都可以追溯到最初的安全需求，通过审核和测试数据汇总成“安全案例（Security case）”。

## 安全保障

**安全保障 (Security Assurance)：** 目的是建立信心。

* **安全案例（Security Case）的三个核心要素：**

1. **安全声明 (Claims)：** 你声称系统具备哪些安全属性。
2. **论证 (Arguments)：** 将你的安全声明串联起来的逻辑链条。
3. **证据 (Evidence)：** 支持上述论证的实质材料（如代码审查报告、形式化证明结果等）。

## 安全风险分析

**安全风险分析 (Security Risk Analysis)：** 列举了微软的经典威胁建模步骤（STRIDE模型的基础）：

1. 识别资产。
2. 创建架构概览。
3. 应用拆解（模块化分析）。
4. 识别威胁。
5. 记录威胁。
6. 对威胁进行评级（优先处理高风险威胁）。

## 系统可信度 (System Trustworthiness)

“可信度”的定义及其评估方式：

* **什么是信任 (Trust)？** 它是指软件组件和利益相关者之间能够**相互依赖**的信心水平。
* **如何实现可信？**
    * **可追溯性：** 必须使用客观、可量化的技术来评估安全需求，且这些评估必须与“安全案例”中的证据链保持一致。
    * **可接受的证据：** 所有的安全证明证据必须让所有利益相关者感到信服。
    * **历史数据：** 大多数信任指标并非凭空产生，而是基于过去在类似信任场景下的行为数据（历史表现）进行评估。
